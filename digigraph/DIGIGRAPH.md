# DigiGraph – Orchestration Brain

**Part of [DigiThings](https://github.com/digithings-ai/digithings) (digithings.ai).**  
**Purpose** (from root `DIGI.md`): Stateful LangGraph core that defines layered agent families, routing, persistent memory, and controlled autonomy.

**Key Patterns (2026 standard)**
- Supervisor + sub-graph architecture
- Layered families (Research Supervisor → Data Science Family → Strategy Generator → Execution Monitor)
- GraphRAG with Graphiti (temporal bi-model memory)
- LiteLLM router (100+ providers, full feature passthrough)
- All outputs Pydantic-structured

**MCP-First Principle**  
Every tool and every agent family has an MCP entrypoint. **Phase 1 orchestrator tools** are registered in-process (`digigraph/orchestration/builtin.py`, optional setuptools entry points `digigraph.tools` in `orchestration/plugins.py`). **DigiSearch** and **DigiQuant** own the canonical OpenAI-style definitions for their vertical tools (`POST /v1/orchestrator_tools`); DigiGraph **fetches** those schemas (cached per index config / base URL) and **dispatches** execution with `POST /v1/orchestrator_invoke` on the same service. `digigraph/tools/digisearch.py` retains a thin `POST /query` client for non-orchestrator call sites (e.g. research utilities). **Remote enumeration of arbitrary third-party MCP servers** remains Phase 2.

**State and Checkpointing**  
Shared agent state carries conversation identity, research outputs, optional `allowed_tool_names` (when a tool allowlist is in effect), Digistore refs, and errors. LangGraph checkpoints persist this state (memory, SQLite, or PostgreSQL). **Auth-bound checkpoints and per-key RBAC** are Phase 2; today **DigiKey JWTs** protect HTTP routes (`DIGIKEY_JWKS_URL` or `DIGIKEY_PUBLIC_KEY_PEM`), plus network isolation, optional tool allowlists, and opt-in HTTP routes (see **digigraph/docs/SECURITY.md**).

**Tool Adapter Pattern**  
Orchestrator tools register in a central registry (`ToolContext`, `get_tools`, `execute`). **Phase 1:** when `ToolContext.allowed_tool_names` is set, `get_tools` only returns those tools and `execute` rejects anything else with a structured error. Sources: request `allowed_tools`, project `agents.allowed_tools`, or env `DIGI_ALLOWED_TOOLS`.

**Handoff and Multi-Agent Coordination**  
Specialist subgraphs (e.g. Data Science Family, Strategy Generator) are invoked by the supervisor via a clear handoff protocol: the supervisor chooses the next agent, updates state (current agent, history), and passes control. Specialists return to the supervisor for routing to tools or the next agent, keeping failures and complexity isolated within subgraphs.

**Agent Activation and Observability**  
Which agent families are active is config-driven (e.g. enabled families and agents). Feature flags and gradual rollout support safe migration of graph or node changes. Observability follows 2026 norms: tracing via **DigiSmith** (`digismith/`) — LangSmith `traceable` wrappers around LLM calls when `LANGSMITH_API_KEY` is set and `langsmith` is installed — plus performance utilities (caching, batching, rate limiting), and cost-per-run visibility for FinOps. See **digismith/DIGISMITH.md**; Docker exposes a DigiSmith status service on port **8003**.

**Agent Family Template** (copy for new families)
```python
# digigraph/agents/<family>/supervisor.py
from langgraph.graph import StateGraph
# ... specialists + supervisor node
```

**Persistent Memory**  
LangGraph checkpoints + Neo4j + Graphiti. Every strategy, backtest, and macro event is stored with temporal validity. Short-term scratchpad state lives in the graph; long-term, strategy-relevant decisions and results are written to the shared strategy memory graph so future runs stay coherent.

**MCP Exposure**  
DigiGraph exposes MCP tools for **hub** workflows (`workflow`, `chat`, `list_orchestrator_tools`, …). Quant **ordering** (backtest → optimize → export) is increasingly owned by **DigiQuant** (`digiquant_run_pipeline`, `POST /v1/workflow`); search **agentic turns** by **DigiSearch** (`digisearch_research_turn`). DigiClaw may attach to the **hub only** or to **vertical MCP servers** directly depending on deployment (see digiclaw docs).

**Federated hub mode (`DIGI_HUB_MODE`)**  
- `legacy` (default): current compiled workflow graph (research subgraph → validate → backtest → optimize). **All** DigiSearch/DigiQuant orchestrator tools still use vertical-owned manifests + invoke HTTP (`/v1/orchestrator_*`); `legacy` only means the **extra** delegate tool names are not registered.  
- `federated`: also registers **`digisearch_research_delegate`** and **`digiquant_pipeline_delegate`** (schemas from the same vertical manifest endpoints) for explicit hub-style delegation. Use project allowlists / `DIGI_ALLOWED_TOOLS` to expose only what each tenant needs.

**Integration Points**  
- Receives calls from digiclaw/  
- Delegates heavy compute to digiquant/  
- Writes to shared strategy memory graph

**Platform (`digibase`)**  
Install **`pip install -e ./digibase`** before `digigraph`. The app registers shared FastAPI error handlers (`{"error": {code, message, request_id, ...}}`), uses `digibase.http.outbound_request_id_headers` toward DigiQuant/DigiSearch, redacts audit payloads via `redact_mapping`, and optionally exports OTel when `OTEL_EXPORTER_OTLP_ENDPOINT` is set and `digibase[otel]` is installed.

**Policy** (`digigraph/policy.py`) centralizes tool allowlists, optional code execution (`DIGI_ALLOW_CODE_EXEC`), and debug/thread gates so middleware and tools share one evaluation path.

**Tool plugins** — After built-ins load, `digigraph/orchestration/plugins.py` loads setuptools entry points **`digigraph.tools`**: callables that return `(name, schema, handler, tags)` tuples for extra orchestrator tools without editing `builtin.py`.

**DigiQuant jobs** — The backtest node prefers **`POST /v1/jobs/backtest`** and polls **`GET /v1/jobs/{id}/status`**, then reuses **`GET /backtest/{id}/result`**. If `/v1/jobs/backtest` is missing (older API), it falls back to `/backtest/start` + SSE, then **`POST /run_backtest`**. Workflow state exposes **`backtest_job_id`** and optional **`quant_artifact_uri`** for opaque result refs (not large blobs).

## Phase 1 implementation (current)

- **Graph:** `digigraph/graph/` — `StateGraph(WorkflowState)`: optional **supervisor** (`DIGI_SUPERVISOR=1`) → **research** as a **compiled subgraph** → conditional **validate_strategy** → optional **backtest** → optional **optimize** → END. **Optimize** runs when `optimize` is in `agents.enabled` or `DIGI_GRAPH_OPTIMIZE_AFTER_BACKTEST=1`, and only after a successful backtest. **Profiles:** YAML `graph.workflow_profile` or `DIGI_WORKFLOW_PROFILE`: `full_stack` (default), `research_rag` (no backtest path), `quant_backtest` / `plan_execute` (same graph shape; combine with `agents.enabled` and `agents.planning_mode`). Document-only turns (freeform `research_response` without `strategy_name`) skip the quant path automatically.
- **Nodes:**
  - **research** (subgraph): **research_inner** runs the Data Science Family: LLM to extract JSON with **`strategy_name`**, **`symbols`**, and optional **`strategy_params`** on the quant path; in **document mode** (project config `research_system_prompt` set) with DigiSearch available: RAG + tool loop and aggregated **`rag_sources`** on state. **research_brief_builder** then emits a typed **`ResearchBrief`** (`digigraph/research_brief_models.py`): themes with **source_ids**, contradictions, assumptions, profiling hooks, and optional catalog strategy hints. Optional **`DIGI_EXTRACT_STRATEGY_AFTER_DOCUMENT_RAG=1`** fills **`strategy_name` / `symbols`** from the brief (or a legacy JSON extract) after the brief step. **`WorkflowRequest.research_filters`** and **`evidence_tier_preference`** merge into every **`digisearch`** / **`digisearch_fetch_all`** call. Tool results still include **`rag_sources`**; streaming emits **trace** events (`graph_update` with brief, **`rag_sources`**, `code_block`, …); OpenAI SSE chunks may include `delta.digigraph_trace` for DigiChat.
  - **validate_strategy**: asserts `strategy_name` and non-empty `symbols` before DigiQuant when the workflow is on the backtest path.
  - **backtest**: DigiQuant via versioned async jobs when available (`/v1/jobs/*`) or legacy `/backtest/start` + `/run_backtest`; writes result, optional `backtest_job_id`, or error into state.
- **LLM:** `digigraph/llm.py` — `get_client()` and `chat_completion()` using `OPENAI_API_BASE` and an API key resolved as **`LITELLM_PROXY_API_KEY`** (if set) else **`OPENAI_API_KEY`** (Bearer for LiteLLM or direct OpenAI), with per-request override from header **`X-LiteLLM-Proxy-Key`** (DigiKey **`litellm_proxy_api_key`** via DigiChat). Tracing uses **`digismith.trace.traceable`** when `LANGSMITH_API_KEY` is set and `langsmith` is installed (see **digismith/DIGISMITH.md**). Local editable installs: `pip install -e ./digismith[langsmith] -e ./digigraph` before `digigraph` alone (monorepo). `get_model_for_mode()` reads **`DIGI_MODEL_MODES_FILE`** under **`DIGI_CONFIG_PATH`** (default filename `model_modes.yaml`) and `DIGI_LLM_MODE` (test|medium|best). When `DIGI_PROJECT_CONFIG` is set, `agents.llm_mode` in the project YAML overrides `DIGI_LLM_MODE`. **Host Ollama:** if `OPENAI_API_BASE` points at Ollama’s OpenAI shim (**`:11434`**), `resolve_effective_model()` strips a leading **`ollama/`** on the configured id (YAML uses LiteLLM-style `ollama/qwen3:8b`; Ollama expects `qwen3:8b`). Override anytime with **`OLLAMA_MODEL`**. Set **`DIGI_DIRECT_OLLAMA_OPENAI=1`** if you use a non-default Ollama port but still need the strip. In Docker, DigiGraph typically uses LiteLLM at `http://litellm:4000/v1`; LiteLLM routes to Ollama Cloud, local Ollama, or OpenAI per config.
- **Project config:** Optional per-project config is loaded via `DIGI_PROJECT_CONFIG` (path to a YAML file; local-only, not in repo). When set, it can specify index config, skills, and `run_data_dir`. Optional `run_data_dir` (or env `DIGI_RUN_DATA_DIR`) enables run storage and delegate tools (visualization_agent, analysis_agent, data_prep_agent).
- **Entrypoint:** `POST /workflow` invokes `run_digigraph_workflow(req)` with `{ "prompt", "session_id", "request_id"?, "allowed_tools"?, "trading_profile"?, "strategy_params"? }`. **`trading_profile`** merges into **`optimization_constraints`** for DigiQuant when present; **`strategy_params`** skips LLM param extraction when you already have a concrete param dict. The server sets `request_id` from the `X-Request-ID` header when the body omits it; that id is propagated to DigiQuant and DigiSearch outbound calls and appears in **`workflow_start` / `workflow_end` / `tool_denied`** audit events (along with a per-run **`workflow_id`**). **Opt-in routes** (Compose defaults `0`; set `1` in `.env` for local dev): `DIGI_ENABLE_DEBUG_ENDPOINTS=1` enables `GET /test_llm` and `GET /v1/debug/*`; `DIGI_ENABLE_THREAD_API=1` enables `GET/POST /threads/*` and `GET /files/*`. **Thread state:** With a checkpointer, `GET /threads/{thread_id}/state` returns checkpoint values; `GET /threads/{thread_id}/history` lists checkpoints. **Human-in-the-loop:** `DIGI_INTERRUPT_AFTER_RESEARCH=1` then `POST /threads/{thread_id}/resume`. See **digigraph/docs/SECURITY.md**.
- **OpenAI-compatible API (Open WebUI):** `GET /v1/models`, `POST /v1/chat/completions`. When `stream: true`, the stream includes tool-call and tool-result blocks; the implementation runs the workflow in a **background thread** feeding SSE (see **digigraph/docs/SECURITY.md** for limits). Optional **Open WebUI format** (`openwebui_format: true`, model `sitaas-rag`, or `X-Response-Format: openwebui`). **Tool allowlist:** request body `allowed_tools` or header `X-Allowed-Tools` (comma-separated). Non-stream responses include `usage` with `estimated: true` (whitespace-split placeholders, not provider-reported tokens). See **digigraph/docs/OPENWEBUI.md**. **DigiChat** (production: **[digichat/](digichat/)** — Next.js BFF + React, Docker profile `digichat`) proxies browser traffic to DigiGraph with **short-lived JWTs** from DigiKey (or a configured static upstream key). Legacy static demo: **website/digichat/**. See **DIGICHAT.md** and **digichat/README.md**.
- **LiteLLM:** Wired in default stack (image pin and health in **DOCKER.md**). DigiGraph depends on LiteLLM; model list in `config/litellm.yaml`, mode defaults in `config/model_modes.yaml`. See **config/MODELS.md** for adding/updating models, caching, and fallbacks.
- **OpenAI Responses API (future):** Chat Completions via LiteLLM remains the default path. When OpenAI Responses-only features are required, plan a thin second client against LiteLLM’s **`/v1/responses`**-compatible proxy surface without breaking multi-provider routing — see [LiteLLM Response API](https://docs.litellm.ai/docs/response_api).
- **Run storage & Digistore:** When `run_data_dir` is set, search results are written to session-scoped storage. **Digistore** (`digigraph/digistore.py`) provides named datasets under `{run_data_dir}/{session_id}/datasets/` with `digistore_put`, `digistore_get`, `digistore_list`, `digistore_profile`. The `digisearch` and `digisearch_fetch_all` tools write results into Digistore (e.g. `search_1`, `search_2`). Use `resolve_dataset_ref(session_id, dataset_ref)` to resolve paths or logical names. **LangGraph persistence:** Optional `stored_datasets` in `WorkflowState` holds refs and short profiles (row_count, columns) so dataset lists survive across turns when the graph is compiled with a checkpointer. **Checkpointer:** unset → **`memory`** (`MemorySaver`); or `sqlite` (file: `DIGI_CHECKPOINTER_SQLITE_URI`, default `~/.digigraph/checkpoints.sqlite`; optional dep `langgraph-checkpoint-sqlite`), or `postgres` (require `DIGI_CHECKPOINTER_POSTGRES_URI`; optional dep `langgraph-checkpoint-postgres`). `DIGI_CHECKPOINTER=none` disables checkpointing (breaks multi-turn). A shared checkpointer is used so `thread_id` persists across HTTP requests; invoke with `config={"configurable": {"thread_id": session_id}}`. The `stream_callback` in state is not serialized—streaming is request-scoped only. See **digigraph/docs/LANGGRAPH_REVIEW.md**.
- **Iterative search:** For "all" result sets (e.g. all emails from user X), use **digisearch_fetch_all**: it paginates automatically and returns a single combined `dataset_ref`. The research prompt and tool descriptions guide the model to use filters (fromAddress, etc.) and fetch_all when appropriate.
- **Sub-agents:** When run storage is enabled, the research node exposes `visualization_agent`, `analysis_agent`, `data_prep_agent`, **data_manipulation_agent**, and **data_engineer_agent**. When the LLM returns **multiple tool calls** in one turn, those that are all **parallel-safe** (tagged `parallel_safe` in the registry, e.g. delegate agents) are executed in parallel to reduce latency; otherwise execution remains sequential. Tool results are appended to the conversation in the same order as the tool calls. The **todo** tool (sitaas_rag) lets the LLM emit an ordered list of intended tasks (e.g. "search all from X", "chart by date", "export CSV"); execution continues with the normal tool loop and the plan is recorded in state for interpretability. The **create_plan** tool submits a structured plan (steps with id, agent, args, depends_on); when **planning_mode** is enabled in project config (`agents.planning_mode: true`), the **executor** (`digigraph/planning/executor.py`) runs the plan: topo-sort by depends_on, resolve placeholders like `{{1.dataset_ref}}` from prior step results, and run independent steps in parallel. A final synthesis LLM turn summarizes the results. See **projects/README.md** and project YAML for `planning_mode`. **Data manipulation** does column math, round, transform, group/aggregate, merge/join, and append via Polars; outputs are written to Digistore. **Data engineer** runs sandboxed Python (Polars only) on one or more datasets; code receives `df_0`, `df_1`, ... and must set `result` to a DataFrame. **Visualization** includes ECharts tools (echarts_line, echarts_bar, echarts_scatter, echarts_pie, echarts_from_code) that return `echarts_option` JSON for frontend rendering with `echarts.init().setOption()`. If Node.js and `echarts` are installed in `digigraph/.../echarts/` (`npm install` there), ECharts results are also converted to SVG and returned as `image_path` so Open WebUI shows the chart inline. Full tool list: **digigraph/docs/OPENWEBUI.md**.
- **MCP / Graphiti:** `python -m digigraph.mcp_server` exposes `workflow`, `chat`, `thread_state`, `list_orchestrator_tools`, and `list_orchestrator_tools_detailed` (install: `pip install -e "digigraph[mcp]"`). The `workflow` tool sets a fresh `request_id` per invocation for audit correlation. Trust model and HTTP gates: **digigraph/docs/SECURITY.md**. **Graphiti** and full DigiClaw subgraph exposure remain Phase 2.

### Orchestration: tools, agents, skills

Tools and agents are organized in three layers so the research node stays decoupled and extensible.

1. **Primitives** — Stateless callables under `digigraph/tools/` (e.g. `tools/digisearch.py` for direct `POST /query`, `tools/analytics/`). Orchestrator **schemas** for search/quant live on the verticals; handlers in `builtin.py` call `vertical_orchestrator` HTTP clients.
2. **Orchestrator tools** — The surface the research node sees: **name**, OpenAI **schema**, **handler** `(args, context) -> result`, and optional **tags** (e.g. `delegate`, `parallel_safe`). Handlers receive a `ToolContext` (session_id, run_data_dir, index_name, index_config, state, optional **allowed_tool_names**, optional **request_id** / **workflow_id** for tracing).
3. **Skills** — Named bundles of orchestrator tool names with an optional **when** predicate. The registry resolves `get_tools(skill_ids, context)` to the list of tool dicts for the LLM; only tools from skills whose `when(context)` is true are included.

**Registry and built-ins:** `digigraph/orchestration/registry.py` defines `ToolContext`, `register_tool`, `register_skill`, `get_tools`, `execute`, `list_tool_names`, and `list_registered_tools_detailed()` (manifest: name, tags, dynamic schema flag). `register_tool` rejects duplicate names. `digigraph/orchestration/builtin.py` registers all built-in tools (digisearch, digisearch_fetch_all, visualization_agent, analysis_agent, data_prep_agent, data_manipulation_agent, data_engineer_agent) and two skills: **search** (digisearch + digisearch_fetch_all, when DigiSearch URL is set) and **sitaas_rag** (search tools + all delegate agents, when `run_data_dir` is set). The **skills library** (`digigraph/skills/`) provides `get_tools_for_skills(skill_ids, context)` (delegating to the orchestration registry) and optional `Skill` metadata (id, name, description). The research node builds `ToolContext`, calls `get_tools_for_skills(cfg.get_enabled_skills(), context)` for the tool list, and uses `execute(name, args, context)` for dispatch. Formatters use `list_tool_names(tag="delegate")` so new delegate agents get correct formatting without code changes.

**Project config:** Optional `skills.enabled` in project YAML can override the default skill list (default is `["search", "sitaas_rag"]`). See `DigiProjectConfig.get_enabled_skills()`. To enable or disable skills per project, set e.g. `skills.enabled: [search]` (search only) or `skills.enabled: [search, sitaas_rag]` in the project config file (see **projects/README.md**).

**Adding a new orchestrator tool (e.g. a new agent):**
1. Add the agent under `digigraph/agents/<name>/`: `schema.py` (OpenAI function tool dict) and `runner.py` (e.g. `run_<name>_agent(...)`).
2. In `digigraph/orchestration/builtin.py`, register the tool: `register_tool("my_agent", MY_AGENT_TOOL, _handle_my_agent, tags={"delegate", "parallel_safe"})` and add a handler that calls your runner with args and `context.session_id` (and other context fields as needed).
3. Add the tool name to the **sitaas_rag** skill in `builtin.py` so it is included when `run_data_dir` is set.
4. No changes needed in `graph/nodes.py` or formatters; the registry and `list_tool_names(tag="delegate")` cover discovery and formatting.

**Adding a new primitive:** Implement the function under `tools/` (e.g. `tools/analytics/`). If it is only used by an existing sub-agent, wire it in that agent’s runner. If it should be callable by the orchestrator as a direct tool, register it in `builtin.py` with a schema and handler, and add it to the appropriate skill.
